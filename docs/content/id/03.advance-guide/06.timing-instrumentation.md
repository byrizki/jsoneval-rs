## Aktifkan timing

Atur `JSONEVAL_TIMING` **sebelum thread pertama kali memakai utility timing**:

```bash
JSONEVAL_TIMING=1 cargo run --release --example basic -- zcc
```

PowerShell:

```powershell
$env:JSONEVAL_TIMING=1
cargo run --release --example basic -- zcc
```

Pengecekan environment menganggap nilai apa pun yang ada sebagai aktif. State timing bersifat thread-local, jadi atur environment sebelum process mulai untuk behavior multi-thread yang dapat diprediksi.

## Kontrol terprogram

```rust
use json_eval_rs::{
    clear_timing_data, disable_timing, enable_timing, print_timing_summary, JSONEval,
};

clear_timing_data();
enable_timing();

let mut evaluator = JSONEval::new(schema, None, None)?;
evaluator.evaluate(data, None, None, None)?;
let compact = evaluator.get_evaluated_schema();

print_timing_summary();
disable_timing();
```

`enable_timing()` dan `disable_timing()` memengaruhi thread saat ini. `clear_timing_data()` hanya membersihkan label/durasi terakumulasi pada thread tersebut.

## Output

Saat aktif, timed block menambahkan catatan `(label, duration)`. `print_timing_summary()` menulis daftar kronologis dan `TOTAL` terjumlah ke **stderr**. Redirect secara terpisah bila diperlukan:

```bash
JSONEVAL_TIMING=1 cargo run --release --example basic -- zcc 2> timing.log
```

Label adalah diagnostik level implementasi dan dapat berubah. Jalur core/evaluasi saat ini mencakup parsing schema/data, pembuatan evaluator, penggantian cache/data, batch evaluation, dependent pass, layout resolution, dan evaluated-schema retrieval.

## Tafsirkan dengan hati-hati

- Gunakan `--release` untuk perbandingan.
- Pisahkan parsing schema cold dari evaluasi warm.
- Bersihkan record di antara skenario.
- Benchmark run berulang dengan schema/data menyerupai production.
- Jangan jumlahkan label bersarang sebagai wall-clock time: interval parent dan child tumpang tindih, sehingga `TOTAL` adalah akumulasi kerja diagnostik, bukan selalu elapsed time process.

Timing nonaktif tetap melakukan branch check timing-enabled di sekitar block terinstrumentasi; jangan mengklaim zero overhead mutlak tanpa mengukur target build Anda.

## Profiling lanjutan

Gunakan instrumentasi ini untuk memilih target, lalu verifikasi dengan workload benchmark atau profiler (misalnya Criterion atau `cargo flamegraph`). Jauhkan schema/data sensitif dari log dan lakukan profiling dalam kondisi mendekati deployment.
