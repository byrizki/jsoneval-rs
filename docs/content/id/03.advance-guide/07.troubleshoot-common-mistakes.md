## Kecilkan kegagalan terlebih dahulu

Mulai dari schema, data, context, dan satu hasil harapan yang paling kecil. Pertahankan input asli tidak berubah saat diagnosis.

```rust
let mut evaluator = JSONEval::new(schema, Some(context), None)?;
evaluator.evaluate(data, Some(context), None, None)?;
println!("{}", evaluator.get_evaluated_schema());
```

Lalu tambahkan field, `$evaluation`, rules, layout, dan subform satu per satu. Ini membedakan JSON/struktur schema tidak valid dari hasil operator yang tidak diharapkan.

## Periksa output yang tepat

| Kebutuhan | Getter |
|---|---|
| Evaluated schema compact | `get_evaluated_schema()` |
| Schema compact tanpa `$params` | `get_evaluated_schema_without_params()` |
| Schema resolved dengan layout overlay | `get_evaluated_schema_resolved()` |
| Satu/banyak path terpilih | `get_evaluated_schema_by_path(s)` |

Output compact memang mempertahankan `$params` dan referensi layout. Jangan mendiagnosis layout gabungan yang hilang dari output compact, dan jangan gunakan argumen boolean getter usang.

## Kesalahan data dan context

- Berikan JSON string valid ke API evaluasi Rust; bedakan data dari `context`.
- Kirim data whole-form saat ini bila formula merujuk nilai sibling atau parent.
- Gunakan konvensi dot/schema-pointer path yang diminta API terkait.
- Tangani field hilang/null secara sengaja dengan rules atau default logic, bukan mengandalkan coercion implisit.
- Untuk output keuangan, tentukan pembulatan secara eksplisit di logic schema dan uji nilai batas.

## Update dependen dan selektif

Setelah perubahan user diketahui, pilih API berdasarkan kebutuhan UI:

```rust
let changed = vec!["applicant.age".to_owned()];
let patches = evaluator.evaluate_dependents(
    &changed,
    Some(data),
    None,
    true,
    None,
    None,
    true,
)?;
```

Gunakan `evaluate(..., Some(&paths), ...)` untuk evaluasi terpilih, `validate(..., Some(&paths), ...)` untuk rule terpilih, dan `evaluate_dependents` saat membutuhkan patch nilai dependen. Pastikan aplikasi patch memakai scope payload sama dengan method yang menghasilkan patch tersebut.

## Subform

Subform harus ada pada array schema path yang dipilih. Periksa sebelum memanggil:

```rust
if evaluator.has_subform("riders.1") {
    evaluator.evaluate_subform("riders.1", parent_data, None, None, None)?;
}
```

Path berindeks (`.1`, `/1`, `[1]`) berjalan dalam scope item array induk dan memproyeksikan patch dependen hasil ke ref item lokal. Panggilan tanpa index memakai data wrapper subform lokal. Lihat [Subforms](use-subforms).

## Kejutan operator

Jangan menganggap JavaScript, Excel, atau engine JSON Logic lain memiliki behavior coercion, date, indexing, empty-array, atau null yang identik. Verifikasi tiap ekspresi dengan test `compile_and_run_logic` minimal dan lihat [referensi operator](/operators/summary) yang sesuai.

Berikan perhatian khusus pada: urutan argumen, scope `var` di dalam operator array, penanganan null/default, konversi/pembulatan numerik, convention parsing/unit tanggal, serta perbandingan strict versus coercing.

## Masalah binding/runtime

- Gunakan hanya package publik: `@json-eval-rs/node`, `@json-eval-rs/bundler`, `@json-eval-rs/vanilla`, atau `@json-eval-rs/react-native`.
- Vanilla memerlukan `await JSONEval.initWasm(...)`; wrapper instance mungkin memerlukan `await evaluator.init()` sebelum digunakan.
- Free/dispose resource evaluator native atau WASM sesuai lifecycle binding.
- Untuk iOS, jalankan `pod install` dan rebuild setelah dependency native berubah; Expo Go tidak dapat memuat native module ini.

## Kumpulkan bukti

Sertakan schema/data/context (rahasia dihapus), package/runtime dan versinya, path/API call tepat, hasil/error, serta reproduksi minimal. Tambahkan `JSONEVAL_TIMING=1` hanya setelah correctness terkonfirmasi saat menyelidiki performa. Lihat [Instrumentasi](timing-instrumentation).
