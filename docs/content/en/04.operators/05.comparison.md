All comparison operators are binary and return a JSON boolean.

## Equality

| Operator | Evaluation rule |
|---|---|
| `==` | JavaScript-like loose equality implemented by JSONEval |
| `!=` | Negated loose equality |
| `===` | Strict JSON value equality |
| `!==` | Negated strict JSON value equality |

```json
{"==": [{"var":"input"}, 1]}
{"===": [{"var":"status"}, "active"]}
{"!==": [{"var":"optional"}, null]}
```

Loose equality handles scalar cross-type cases: numbers and strings compare via numeric parsing; booleans compare with numbers/strings through numeric conversion. `null` equals only `null` in the current implementation. Other values fall back to strict JSON-value equality.

Use strict equality for data-contract checks. Use loose equality only when the accepted input representation deliberately includes equivalent scalar forms, and test those forms.

## Ordering

| Operator | Condition |
|---|---|
| `<` | left is less than right |
| `<=` | left is less than or equal to right |
| `>` | left is greater than right |
| `>=` | left is greater than or equal to right |

Ordering uses JSONEval numeric comparison (`to_f64(left) - to_f64(right)`), not lexical string ordering. Validate/normalize user-facing strings and dates before using them as ordered values.

```json
{"<": [{"var":"age"}, 18]}
{"and": [
  {">=": [{"var":"amount"}, {"var":"minimum"}]},
  {"<=": [{"var":"amount"}, {"var":"maximum"}]}
]}
```

## Common patterns

### Presence plus range

```json
{"and": [
  {"!==": [{"var":"age"}, null]},
  {">=": [{"var":"age"}, 18]},
  {"<=": [{"var":"age"}, 65]}
]}
```

### Exact discriminator

```json
{"if": [
  {"===": [{"var":"account.type"}, "business"]},
  "business-flow",
  "consumer-flow"
]}
```

### Numeric calculation boundary

```json
{"<=": [
  {"round": [{"var":"computedTotal"}, 2]},
  {"var":"creditLimit"}
]}
```

Decide and test whether rounding happens before comparison; this changes financial/business outcomes.

## Avoid unsupported pseudo-operators

The core compiler provides the eight operators on this page. Do not introduce `let`, `ToNumber`, date-addition, or custom coercion expressions in a schema unless the project separately provides and tests them.
