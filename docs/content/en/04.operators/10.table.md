A table is an array of JSON objects. Most search operators return a zero-based index or `-1` when no row matches; `VALUEAT` then returns `null` for that invalid index. Guard a lookup if `null` is not an acceptable downstream value.

## Direct access

| Operator | Form | Result |
|---|---|---|
| `VALUEAT` | `[table, rowIndex, column?]` | Row or column value; `null` for invalid index/column |
| `MAXAT` | `[table, column]` | Column value of final row; `null` for empty/missing data |

```json
{"VALUEAT": [{"var":"rates"}, 1, "premium"]}
{"MAXAT": [{"var":"rates"}, "age"]}
```

Indexes are zero-based. `MAXAT` means *last row access*, not numeric maximum; sort/select table rows before using it as a maximum shortcut.

## Exact and range index lookup

`INDEXAT` takes `[lookupValue, table, field, range?]`.

- Exact mode is default, uses loose equality, and returns first matching row index.
- Truthy `range` mode returns the **first** row whose numeric field value is `<= lookupValue`.

```json
{"INDEXAT": [{"var":"id"}, {"var":"users"}, "id"]}
{"INDEXAT": [{"var":"age"}, {"var":"brackets"}, "minimumAge", true]}
```

Range mode does not choose the greatest eligible threshold. It searches table order and stops at first eligible row. Order the table to match that contract, or use explicit `MATCHRANGE` bounds.

## Multi-field lookup

`MATCH` takes table plus `[value, field]` pairs and returns the first row where **all** valid pairs match using loose equality.

`CHOOSE` has the same pair syntax but returns the first row where **any** pair matches.

```json
{"MATCH": [{"var":"rates"}, "ID", "country", "standard", "plan"]}
{"CHOOSE": [{"var":"rates"}, "ID", "country", "standard", "plan"]}
```

Use `MATCH` for an intersection of criteria. Use `CHOOSE` only when one criterion is sufficient. Do not rely on claims that it randomly chooses a row.

## `MATCHRANGE`

`MATCHRANGE` takes table plus `[minField, maxField, value]` triplets. It returns first row where every triplet satisfies inclusive numeric bounds: `min <= value <= max`.

```json
{"MATCHRANGE": [
  {"var":"rateTable"},
  "minAge", "maxAge", {"var":"age"},
  "minTerm", "maxTerm", {"var":"term"}
]}
```

Missing bound fields convert through the evaluator numeric conversion, so validate table rows at schema/data boundaries.

## `FINDINDEX`

`FINDINDEX` takes `[table, condition...]` and returns first row satisfying all conditions. With no conditions it returns `0` for a non-empty table.

The compiler supports concise table conditions:

```json
{"FINDINDEX": [{"var":"users"}, "active"]}
{"FINDINDEX": [{"var":"users"}, [">=", 18, "age"]]}
{"FINDINDEX": [
  {"var":"users"},
  ["==", "active", "status"],
  {">=": [{"var":"score"}, 70]}
]}
```

- A standalone string becomes a truthiness check on that row field.
- A comparison triplet is `[operator, value, field]`; its field is read from the row.
- Object conditions can use normal expressions. Bare `var` reads resolve from internal context, row, then outer data; `$ref` reads outer/internal context rather than the row.

## Safe retrieval pattern

```json
{"if": [
  {">=": [
    {"MATCHRANGE": [{"var":"rates"}, "minAge", "maxAge", {"var":"age"}]},
    0
  ]},
  {"VALUEAT": [
    {"var":"rates"},
    {"MATCHRANGE": [{"var":"rates"}, "minAge", "maxAge", {"var":"age"}]},
    "rate"
  ]},
  null
]}
```

If repeated lookup costs matter, compute/emit its result once in schema logic rather than introducing unsupported `let` syntax.

## Test boundaries

Test empty table, `-1` lookup result, absent column, duplicate rows, exact loose-equality inputs, `INDEXAT` order, inclusive `MATCHRANGE` endpoints, and `FINDINDEX` row versus outer-data references. See [Array](array) for collection operations.
