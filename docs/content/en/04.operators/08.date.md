Date operators parse supported date strings into calendar dates. Invalid or non-string date input generally produces `null`. Use unambiguous ISO input (`YYYY-MM-DD`) in schemas and data.

## Current date/time and components

| Operator | Result |
|---|---|
| `today`, `TODAY` | Current calendar date as ISO date text |
| `now`, `NOW` | Current RFC 3339 timestamp, seconds precision |
| `year`, `YEAR` | Calendar year or `null` |
| `month`, `MONTH` | Month from 1 through 12 or `null` |
| `day`, `DAY` | Day of month or `null` |

```json
{"TODAY": null}
{"YEAR": {"var":"birthDate"}}
{"and": [
  {"===": [{"MONTH": {"var":"date"}}, 12]},
  {"===": [{"DAY": {"var":"date"}}, 25]}
]}
```

`today` and `now` apply an evaluator timezone offset when configured; otherwise they use UTC. Date component extraction follows the evaluator's date parsing/configuration path. Test dates near midnight with the same evaluator settings used in production.

## `date` / `DATE`

Constructs an ISO date from `[year, month, day]`. Month/day overflow and underflow normalize in a JavaScript-compatible way.

```json
{"DATE": [2024, 12, 25]}
{"date": [2024, 13, 1]}
{"date": [2024, 3, 0]}
```

The latter two normalize into adjacent calendar periods. Validate user-supplied components before construction if normalization is not acceptable business behavior.

## `days` / `DAYS`

Computes whole calendar days as `[endDate, startDate]`; a reversed order produces a negative result.

```json
{"DAYS": [{"var":"endDate"}, {"var":"startDate"}]}
```

## `datedif` / `DATEDIF`

Computes `[startDate, endDate, unit]`. Units are case-insensitive:

| Unit | Result |
|---|---|
| `D` | Total day difference |
| `M` | Completed whole months |
| `Y` | Completed whole years |
| `MD` | Day remainder using the implementation's simplified 30-day month rule |
| `YM` | Month remainder after years |
| `YD` | Day difference after aligning year |

```json
{"DATEDIF": [{"var":"birthDate"}, {"TODAY": null}, "Y"]}
```

Use `Y` for completed-age-style results. Treat `MD`, `YM`, and `YD` as application-specific calendar calculations: cover leap dates and month ends with fixtures.

## `yearfrac` / `YEARFRAC`

Computes `[startDate, endDate, basis?]`. Current basis handling is:

| Basis | Divisor |
|---|---|
| `0`, `2`, `4` | 360 |
| `1` | 365.25 |
| `3` or unsupported value | 365 |

```json
{"YEARFRAC": [{"var":"periodStart"}, {"var":"periodEnd"}, 1]}
```

This is a fixed-day-divisor calculation, not a complete implementation of named financial day-count conventions. Validate required contractual convention with representative cases.

## `dateformat` / `DATEFORMAT`

Formats `[date, format?]`; default format is `iso`. Built-ins are `short`, `long`, `iso`, `us`, `eu`, `full`, `monthday`, `yearmonth`, `ddmmyyyy`, `mmddyyyy`, `yyyymmdd`, `dd-mm-yyyy`, `mm-dd-yyyy`, `yyyy-mm-dd`, and `dd.mm.yyyy`. Other strings are passed to Chrono-style `strftime` formatting.

```json
{"DATEFORMAT": [{"var":"date"}, "dd/mm/yyyy"]}
{"dateformat": [{"var":"date"}, "%Y-%m-%d"]}
```

Named `yyyy` formats are predefined names; custom formats use `%Y`, `%m`, and `%d`, not `YYYY-MM-DD`.

## Test boundaries

Test invalid input, leap day, month end, order of date differences, timezone offset near midnight, and every financial basis your product accepts. See [Comparison](comparison) for predicates and [Math](math) for numeric rounding.
