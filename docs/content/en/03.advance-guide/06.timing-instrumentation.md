## Enable timing

Set `JSONEVAL_TIMING` **before the thread first uses timing utilities**:

```bash
JSONEVAL_TIMING=1 cargo run --release --example basic -- zcc
```

PowerShell:

```powershell
$env:JSONEVAL_TIMING=1
cargo run --release --example basic -- zcc
```

The environment check treats any present value as enabled. Timing state is thread-local, so set the environment before process startup for predictable multi-thread behavior.

## Programmatic control

```rust
use json_eval_rs::{
    clear_timing_data, disable_timing, enable_timing, print_timing_summary, JSONEval,
};

clear_timing_data();
enable_timing();

let mut evaluator = JSONEval::new(schema, None, None)?;
evaluator.evaluate(data, None, None, None)?;
let compact = evaluator.get_evaluated_schema();

print_timing_summary();
disable_timing();
```

`enable_timing()` and `disable_timing()` affect the current thread. `clear_timing_data()` clears only that thread's accumulated labels/durations.

## Output

When enabled, timed blocks append `(label, duration)` records. `print_timing_summary()` writes a chronological list and summed `TOTAL` to **stderr**. Redirect it separately if needed:

```bash
JSONEVAL_TIMING=1 cargo run --release --example basic -- zcc 2> timing.log
```

Labels are implementation-level diagnostics and can change. Current core/evaluation paths include schema/data parsing, evaluator construction, cache/data replacement, batch evaluation, dependent passes, layout resolution, and evaluated-schema retrieval.

## Interpret carefully

- Use `--release` for comparison.
- Separate cold schema parsing from warm evaluation.
- Clear records between scenarios.
- Benchmark repeated runs with production-shaped schema/data.
- Avoid summing nested labels as wall-clock time: parent and child intervals overlap by design, so `TOTAL` is diagnostic accumulated work, not necessarily elapsed process time.

Timing disabled still takes a timing-enabled branch check around instrumented blocks; do not claim absolute zero overhead without measuring your target build.

## More profiling

Use this instrumentation to choose a target, then verify it with a workload benchmark or profiler (for example Criterion or `cargo flamegraph`). Keep sensitive schema/data out of logs and profile in conditions close to deployment.
