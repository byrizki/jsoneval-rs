## Purpose

`ParsedSchemaCache` stores `Arc<ParsedSchema>` values behind `Arc<RwLock<IndexMap<...>>>`. It avoids reparsing a schema revision when application code needs separate evaluators for different data/context state.

Cache ownership and eviction are caller-controlled. `PARSED_SCHEMA_CACHE` is an optional global instance; it does not populate itself when a normal evaluator is constructed.

## Basic lifecycle

```rust
use json_eval_rs::{JSONEval, ParsedSchema, ParsedSchemaCache};
use std::sync::Arc;

let cache = ParsedSchemaCache::new();
let parsed = Arc::new(ParsedSchema::parse(schema_json)?);
cache.insert("tenant:42:v3".to_owned(), parsed);

let parsed = cache
    .get("tenant:42:v3")
    .ok_or("schema was not loaded")?;
let mut evaluator = JSONEval::with_parsed_schema(parsed, None, None)?;
evaluator.evaluate(data, None, None, None)?;

let compact = evaluator.get_evaluated_schema();
cache.remove("tenant:42:v3");
```

Each `get` returns a cheap `Arc` clone. Existing evaluators remain valid after `remove`/`clear` while they retain their own `Arc` reference.

## Cache operations

```rust
cache.contains_key("tenant:42:v3");
cache.len();
cache.is_empty();
cache.keys();
cache.stats();

let parsed = cache.get_or_insert_with("tenant:42:v3", || {
    Arc::new(ParsedSchema::parse(schema_json).expect("valid schema"))
});

cache.insert_batch(entries);
let removed = cache.remove_batch(&keys);
cache.clear();
```

`insert` replaces and returns any previous value. Choose keys that encode schema ownership and revision, such as `tenant:<id>:<version>`, and remove superseded entries intentionally.

## Concurrency

The cache safely supports concurrent reads and serialized writes. It shares immutable `ParsedSchema` values, but a `JSONEval` constructed from a parsed schema still owns mutable evaluation data, context, output, and caches. Create a separate evaluator per concurrent request/job unless your application adds synchronization.

## Bindings

Core Rust exposes cache population APIs. Binding-level `fromCache`/`FromCache` constructors, where available, require a native cache key that has already been populated by supported native setup. They are not universal JavaScript/C# cache loaders; verify the installed binding before documenting a preload flow.

## Decide whether to cache

Use it for repeated evaluation of the same schema revision, not every small one-off schema. Track key count/memory through `stats`, test schema-reload behavior, and measure end-to-end workload before adopting a global cache.

For MessagePack transport and live evaluator reuse, see [Caching & Performance](caching-and-performance).
