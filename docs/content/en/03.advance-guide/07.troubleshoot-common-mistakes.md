## Reduce the failure first

Start with the smallest schema, data, context, and one expected result. Keep the original input immutable while diagnosing.

```rust
let mut evaluator = JSONEval::new(schema, Some(context), None)?;
evaluator.evaluate(data, Some(context), None, None)?;
println!("{}", evaluator.get_evaluated_schema());
```

Then add fields, `$evaluation`, rules, layout, and subforms one at a time. This distinguishes invalid JSON/schema structure from an unexpected operator result.

## Check the right output

| Need | Getter |
|---|---|
| Compact evaluated schema | `get_evaluated_schema()` |
| Compact schema without `$params` | `get_evaluated_schema_without_params()` |
| Layout-overlaid resolved schema | `get_evaluated_schema_resolved()` |
| One/many selected paths | `get_evaluated_schema_by_path(s)` |

Compact output intentionally retains `$params` and layout references. Do not diagnose a missing merged layout from compact output, and do not use obsolete boolean getter arguments.

## Data and context mistakes

- Provide valid JSON strings to Rust evaluation APIs; distinguish data from `context`.
- Pass current whole-form data when formulas reference sibling or parent values.
- Use the same dot/schema-pointer path convention the relevant API expects.
- Treat missing/null fields deliberately with rules or logic defaults rather than relying on implicit coercion.
- For financial outputs, define rounding explicitly in schema logic and test boundary values.

## Dependent and selective updates

After a known user change, choose the API based on what UI needs:

```rust
let changed = vec!["applicant.age".to_owned()];
let patches = evaluator.evaluate_dependents(
    &changed,
    Some(data),
    None,
    true,
    None,
    None,
    true,
)?;
```

Use `evaluate(..., Some(&paths), ...)` for selected evaluation, `validate(..., Some(&paths), ...)` for selected rules, and `evaluate_dependents` when you need dependent patches. Ensure patch application uses the same payload scope as the method that produced it.

## Subforms

A subform must exist at the selected array schema path. Check it before calling:

```rust
if evaluator.has_subform("riders.1") {
    evaluator.evaluate_subform("riders.1", parent_data, None, None, None)?;
}
```

Indexed paths (`.1`, `/1`, `[1]`) run in parent array-item scope and project returned dependent patches to local item refs. Non-indexed calls use local subform-wrapper data. See [Subforms](use-subforms).

## Operator surprises

Do not assume JavaScript, Excel, or another JSON Logic engine has identical coercion, date, indexing, empty-array, or null behavior. Verify each expression with a minimal `compile_and_run_logic` test and consult the matching [operator reference](/operators/summary).

Pay particular attention to: argument order, `var` scope within array operators, null/default handling, numeric conversion and rounding, date parsing/unit conventions, and strict versus coercing comparisons.

## Binding/runtime issues

- Use only public packages: `@json-eval-rs/node`, `@json-eval-rs/bundler`, `@json-eval-rs/vanilla`, or `@json-eval-rs/react-native`.
- Vanilla needs `await JSONEval.initWasm(...)`; wrapper instances may need `await evaluator.init()` before use.
- Free/dispose native or WASM evaluator resources according to the binding lifecycle.
- For iOS, run `pod install` and rebuild after native dependency changes; Expo Go cannot load this native module.

## Gather evidence

Capture schema/data/context (with secrets removed), runtime package/version, exact path/API call, result/error, and a minimal reproduction. Add `JSONEVAL_TIMING=1` only after correctness is established when investigating performance. See [Instrumentation](timing-instrumentation).
